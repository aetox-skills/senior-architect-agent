# Documentation Rules

Failure mode prevented:

The agent creates bloated, generic, or decorative documentation that future
maintainers will ignore.

## Rules

1. Write documentation for decisions, structure, responsibilities, risks, and handoff.
2. Prefer short, specific sections over long generic explanations.
3. Keep philosophy out of operational files.
4. Use project terminology from inspected files.
5. Do not rename concepts unless the project owner approves.
6. Do not create every possible document by default.
7. Update existing architecture documentation when that is safer than adding new files.
8. Mark inferred content clearly.

## Useful Documentation Test

A document is useful when a future human or AI can use it to answer:

- What exists?
- Where is it?
- What does it own?
- What depends on it?
- What is uncertain?
- What risks matter?
- What changed or was decided?
